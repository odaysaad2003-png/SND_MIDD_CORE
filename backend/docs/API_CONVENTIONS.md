# API Conventions

## Base Path and Versioning

All current endpoints are under:

```text
/api/v1
```

Do not make silent backward-incompatible changes to `/api/v1`. Use a migration plan or a
new API version when clients cannot remain compatible.

## Resource and Path Naming

- Use plural resource nouns: `/posts`, `/comments`, `/reports`.
- Use `posts`, never `listings`.
- Use descriptive parameter names: `:postId`, `:commentId`, `:reportId`.
- Use `/me` for authenticated-user collections/resources when the identity is implicit.
- Use nesting when the parent is required to address the relationship, for example
  `/posts/:postId/comments`.
- Static paths (`/me`, `/stats`) and nested routers must be mounted before generic
  `/:resourceId` routes.
- New broad admin capabilities should use a clear admin namespace when practical; do not
  rename existing routes without a compatibility plan.

## Request Data Placement

- **Path params** identify a resource and must be validated as MongoDB ObjectIds.
- **Query params** control collection reads: pagination, filtering, sorting, and search.
- **JSON body** carries state-changing structured input.
- **Multipart/form-data** carries uploads; file fields and any accompanying text fields
  must be validated explicitly.
- Never accept identity, ownership, role, counters, storage public IDs, or protected status
fields from an untrusted body unless the endpoint is explicitly designed and authorized
for that exact field.

## Content Types

- JSON endpoints require `Content-Type: application/json` when a body is sent.
- Upload endpoints require `multipart/form-data` generated by the client; do not manually
hardcode the multipart boundary.
- `204 No Content` responses contain no JSON body.

## Success Response

```json
{
  "success": true,
  "data": {},
  "meta": {}
}
```

- `data` is required for normal JSON success responses and may be an object, array, or a
small action result.
- `meta` is optional and used for pagination or non-resource metadata.
- Do not wrap the same value in inconsistent keys across equivalent endpoints.

## Error Response

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed",
    "details": [
      { "path": "body.title", "message": "Title is required" }
    ]
  }
}
```

- `code` is stable and machine-readable.
- `message` is safe for clients.
- `details` is optional and normally used for validation information.
- Production responses never expose stack traces, secrets, raw database errors, or vendor
responses.
- A request ID may be exposed in headers or the error payload if the implementation does
so consistently.

## Foundation Utilities

Central behavior is implemented through:

```text
src/utils/app-error.ts
src/utils/async-handler.ts
src/utils/api-response.ts
src/constants/error-codes.ts
src/middleware/error-handler.ts
src/middleware/validate-request.ts
```

Extend these utilities rather than creating module-specific response/error formats.

## HTTP Status Rules

| Status | Use |
|---|---|
| `200 OK` | Successful read/update/action with a response body |
| `201 Created` | A new resource/relation document was created |
| `204 No Content` | Successful delete action that intentionally returns no body |
| `400 Bad Request` | Invalid request semantics or validation failure |
| `401 Unauthorized` | Missing, invalid, or expired authentication |
| `403 Forbidden` | Authenticated but not permitted |
| `404 Not Found` | Resource unavailable under the endpoint's visibility rules |
| `409 Conflict` | Unique constraint or state conflict |
| `413 Payload Too Large` | Upload/body exceeds configured size when supported by middleware |
| `415 Unsupported Media Type` | Unsupported content/file type when supported by the API contract |
| `429 Too Many Requests` | Rate limit exceeded |
| `500 Internal Server Error` | Unexpected server failure |

Do not leak whether a private/deleted resource exists when returning `404` is the safer
contract.

## Pagination

Current collection endpoints use offset pagination:

```text
?page=1&limit=10
```

Rules:

- `page` is a positive integer; default `1`.
- `limit` is a positive integer with a module-approved maximum (currently commonly `50`).
- Service skip is `(page - 1) * limit`.
- Return consistent metadata:

```json
{
  "page": 1,
  "limit": 10,
  "total": 37,
  "totalPages": 4
}
```

A zero-result collection may return `totalPages: 0`; clients must not infer that page 1
contains data. Cursor pagination is a future measured optimization, not a global rewrite.

## Sorting

Use validated, controlled values rather than arbitrary MongoDB fields. Existing list
endpoints commonly use:

```text
?sort=latest
?sort=oldest
```

Implement deterministic sorting; include `_id` as a tie-breaker in aggregation pipelines
when equal timestamps are possible.

## Search and Filtering

- Filter values are explicit allowlisted query fields.
- Search field names already exposed by an endpoint must remain backward compatible.
- For new endpoints, prefer a concise `q` field unless an existing module convention
requires `search`.
- Search strings are trimmed, length-limited, and never inserted into raw query operators
without escaping/validation.
- Never pass `req.query` directly to Mongoose.

## Authentication Transport

Current backend-development contract:

```http
Authorization: Bearer <access-token>
```

Refresh tokens currently use validated JSON request/response fields for Postman testing.
This is not the final browser production contract. The future cookie migration must be a
deliberate API/security change, not a silent controller edit.

## Delete and Toggle Semantics

- Posts/comments use soft-delete behavior defined by their services.
- Like/save routes are idempotent relationship toggles and may return `200` when no new
document was created.
- A first-time relation creation may return `201`; a restore/no-op may return `200`.
- Endpoints returning an updated count/status must use `200`, not `204`.
- Exact idempotency behavior must be documented by the service/controller tests.

## Upload Conventions

- Resource creation and post-image upload remain separate endpoints.
- The server controls file count, size, MIME allowlist, storage path/folder, and public ID.
- API responses expose display URLs, not Cloudinary secrets or internal deletion IDs.
- Partial upload failures require best-effort cleanup of newly uploaded assets.
- Before production launch, validate file signatures/magic bytes in addition to MIME.

## Example — Create Post

```http
POST /api/v1/posts
Authorization: Bearer <access-token>
Content-Type: application/json
```

```json
{
  "title": "Need help moving furniture",
  "content": "Looking for two volunteers tomorrow afternoon."
}
```

The server derives the author and protected fields.

## Example — Paginated Collection

```http
GET /api/v1/posts?page=1&limit=10&sort=latest&search=help
```

```json
{
  "success": true,
  "data": [],
  "meta": {
    "page": 1,
    "limit": 10,
    "total": 0,
    "totalPages": 0
  }
}
```

The exact supported query key (`search` or `q`) remains defined by that route's current
validation schema; do not rename it without compatibility review.
